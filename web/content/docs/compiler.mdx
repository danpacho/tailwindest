> If you do not use `CreateCompiledTailwindest` nested variant shorthand, you
> can use normal Tailwindest runtime authoring.
>
> The compiler is for projects that want raw nested variant leaves in source
> code and exact Tailwind class candidates in the Tailwind CSS build.

`@tailwindest/compiler` is the nested variant lowering layer for Tailwindest.
It lowers `CreateCompiledTailwindest` object paths into Tailwind variant
prefixes and forwards the resulting candidates to Tailwind CSS v4.

It is not a general Tailwindest partial evaluator. Runtime-visible objects,
stylers, `tw.join(...)`, and `tw.mergeRecord(...)` remain runtime API calls.

## What It Does

```tsx
const button = tw.style({
    display: "flex",
    backgroundColor: "bg-red-50",
    dark: {
        backgroundColor: "bg-red-900",
        hover: {
            backgroundColor: "bg-red-950",
        },
    },
})

button.class()
```

When the terminal call produces a class string and the input is exact, compiled
output is equivalent to:

```ts
"flex bg-red-50 dark:bg-red-900 dark:hover:bg-red-950"
```

Unsupported call sites are preserved as Tailwindest runtime calls. Statically
knowable candidates are still retained for the Tailwind CSS manifest.

## Install

```bash
pnpm add tailwindest tailwindcss @tailwindcss/vite vite
pnpm add -D @tailwindest/compiler
```

The compiler targets Vite and Tailwind CSS v4. Tailwind CSS and Vite remain
application-owned dependencies.

## Vite Setup

Place `tailwindest()` before `tailwindcss()`:

```ts filename="vite.config.ts"
import tailwindcss from "@tailwindcss/vite"
import { tailwindest } from "@tailwindest/compiler/vite"
import { defineConfig } from "vite"

export default defineConfig({
    plugins: [
        tailwindest({
            debug: true,
            sourceMap: true,
        }),
        tailwindcss(),
    ],
})
```

The Vite integration returns two coordinated plugins:

- `tailwindest:transform` compiles JavaScript and TypeScript modules.
- `tailwindest:source` injects generated Tailwind candidates into CSS through
  Tailwind CSS v4 `@source inline()`.

This means development, debug, and production builds use the same compiler
manifest instead of relying on Tailwind to discover post-transform JavaScript.
The CSS side also loads Tailwind's design-system variant groups, so compiled
nested shorthand uses the same variant metadata as `create-tailwind-type`.

## Authoring Types

Compiler-oriented projects should use
[`CreateCompiledTailwindest`](/apis/createcompiledtailwindest). It keeps variant
prefixes in object keys and uses raw Tailwind utilities at leaf values.

This type exists because the compiler can derive variant prefixes from the
object path. You should not have to autocomplete or write `dark:hover:` inside
the value when `dark` and `hover` are already present as object keys.

```ts filename="tw.ts"
import {
    createTools,
    type CreateCompiledTailwindest,
    type CreateTailwindLiteral,
} from "tailwindest"
import type { Tailwind, TailwindNestGroups } from "./tailwind"

export type Tailwindest = CreateCompiledTailwindest<{
    tailwind: Tailwind
    tailwindNestGroups: TailwindNestGroups
    useArbitrary: true
}>

export type TailwindLiteral = CreateTailwindLiteral<Tailwind>

export const tw = createTools<{
    tailwindest: Tailwindest
    tailwindLiteral: TailwindLiteral
}>()
```

For the complete type contract, arbitrary variant support, group prefix
behavior, and CSS Transformer integration, see the
[`CreateCompiledTailwindest` API reference](/apis/createcompiledtailwindest).

This path-prefix normalization is compiler-owned. Runtime Tailwindest keeps
object keys structural and preserves the leaf class strings you authored.
The compiler does not use a hard-coded variant allowlist. It derives variant
keys from the active Tailwind CSS design system through the same internal
Tailwind compiler package used by `create-tailwind-type`.

Nested variants should be written like this:

```ts
tw.style({
    backgroundColor: "bg-red-50",
    dark: {
        backgroundColor: "bg-red-900",
        hover: {
            backgroundColor: "bg-red-950",
        },
    },
})
```

Do not write prefixed leaf values in compiler-oriented source:

```ts
tw.style({
    dark: {
        backgroundColor: "dark:bg-red-900",
    },
})
```

The compiler accepts explicit-prefixed leaves for compatibility, but raw nested
leaves are the canonical form for `CreateCompiledTailwindest`.

### Variant Metadata

The Vite plugin loads variant metadata from CSS entries that import Tailwind
CSS. This is what lets the compiler understand project-specific variants such
as `surface`, `data-[state=open]`, custom variants, responsive variants, and
arbitrary variants without expecting a generated `tailwind.ts` file at runtime.

If a low-level programmatic compile cannot load Tailwind CSS metadata, nested
compiled shorthand falls back instead of guessing. Use `compileAsync()` with
`cssRoot` or `cssSource` for fully correct compiled shorthand outside Vite.

## Compile Contract

The compiler has one production behavior: lower nested variant object syntax to
Tailwind class prefixes, then bridge those candidates into Tailwind CSS.

JavaScript replacement is allowed only when the final observable value is a
class string and the compiler can prove the result exactly. Unsupported call
sites are preserved as Tailwindest runtime calls. Diagnostics explain why a
call remained at runtime, and statically knowable Tailwind candidates are still
forwarded to the Tailwind CSS manifest.

There is no compiler policy switch. Development, debug, production, and
programmatic compilation all use the same contract.

Raw nested shorthand is not runtime-compatible in object-returning channels.
For example, this call is preserved, candidates are collected when possible,
and the compiler reports `COMPILED_VARIANT_REQUIRES_CLASS_OUTPUT`:

```ts
tw.style({ dark: { color: "text-white" } }).style()
```

Use runtime-compatible prefixed values when a runtime object or styler value
must cross the compiler boundary:

```ts
tw.style({ dark: { color: "dark:text-white" } }).style()
```

## Options

```ts
tailwindest({
    include: [/src\/.*\.[jt]sx?$/],
    exclude: [/node_modules/],
    debug: true,
    sourceMap: true,
    variantTableLimit: 256,
    cssEntries: [/src\/app\.css$/],
    scanRoots: ["src"],
})
```

| Option                           | Default                            | Description                                                                  |
| -------------------------------- | ---------------------------------- | ---------------------------------------------------------------------------- |
| `include`                        | all JS/TS modules                  | Source id patterns eligible for compilation.                                 |
| `exclude`                        | none                               | Source id patterns never compiled or used as CSS entries.                    |
| `debug`                          | `false`                            | Writes `.tailwindest/debug-manifest.json`.                                   |
| `debugFile`                      | `.tailwindest/debug-manifest.json` | Custom debug manifest path.                                                  |
| `sourceMap`                      | `false`                            | Emits source maps for exact replacements.                                    |
| `variantTableLimit`              | internal default                   | Maximum generated lookup table size for bounded variants.                    |
| `cssEntries`                     | Tailwind CSS imports               | CSS ids that should receive `@source inline()` injection.                    |
| `scanRoots`                      | `["src"]` under Vite root          | Directories scanned before the first CSS transform.                          |
| `merger`                         | `{ kind: "none" }`                 | Build-time class merge policy.                                               |
| `collectStringLiteralCandidates` | `true`                             | Collects standalone candidate-like string literals from transformed modules. |

## Supported Compile Surface

The compiler targets the public `createTools()` API:

| API                                  | Replacement role                                                                      |
| ------------------------------------ | ------------------------------------------------------------------------------------- |
| `tw.style(obj).class(...extra)`      | exact class-output lowering                                                           |
| `tw.mergeProps(...styles)`           | exact class-output helper when every input proves static                              |
| `tw.def(classList, ...styles)`       | exact class-output helper when every input proves static                              |
| `tw.toggle(config).class(condition)` | exact class-output lowering when the condition is proven or a safe ternary is emitted |
| `tw.rotary(config).class(key)`       | exact class-output lowering when the key is proven or a finite lookup is safe         |
| `tw.variants(config).class(props)`   | exact class-output lowering when the bounded table is collision-safe                  |
| `tw.join(...)`                       | candidate collection only; no JS replacement contract                                 |
| `*.style(...)`                       | runtime-visible object value; no JS replacement                                       |
| `*.compose(...)`                     | runtime-visible styler value; no JS replacement                                       |
| `tw.mergeRecord(...)`                | runtime-visible object value; no JS replacement                                       |

Compile eligibility is based on symbol provenance, not variable names. A
receiver named `tw` is not enough; the compiler must prove that it originates
from `createTools()`.

If a runtime merger is configured, class-output replacement is disabled unless
the merger is known to be deterministic at build time. The compiler preserves
the original call and reports a diagnostic instead of approximating merge
semantics.

## Tailwind CSS Bridge

The compiler does not use Tailwind plugin `addUtilities`. Tailwindest generates
candidate class names for Tailwind CSS built-in utilities; it does not define
new CSS utilities.

Generated candidates are injected into CSS with Tailwind CSS v4 `@source
inline()`:

```css filename="app.css"
@import "tailwindcss";

/* tailwindest:start */
@source inline("bg-red-50 dark:bg-red-900 dark:hover:bg-red-950");
/* tailwindest:end */
```

When nested raw leaves would otherwise leak as top-level classes, the compiler
can also inject effective exclusions with `@source not inline()`.

## Debugging

Enable debug mode during integration:

```ts
tailwindest({
    debug: true,
    sourceMap: true,
})
```

The debug manifest records:

- transformed files and hashes
- replacement spans
- generated text
- Tailwind candidates
- effective exclusions
- fallback status
- diagnostics

Inspect `.tailwindest/debug-manifest.json` when a class is missing from the CSS
output or a call unexpectedly falls back.

## Programmatic Compile

Build tools that cannot use the Vite plugin can call `compile()` directly:

```ts
import { compile } from "@tailwindest/compiler"

const result = compile(source, {
    fileName: "/repo/src/button.tsx",
    sourceMap: true,
})

console.log(result.code)
console.log(result.candidates)
console.log(result.diagnostics)
```

`compile()` is file-local and does not load Tailwind CSS design-system metadata.
It is enough for already-prefixed classes and non-nested shorthand, but nested
compiled shorthand needs `compileAsync()`:

```ts
import { compileAsync } from "@tailwindest/compiler"

const result = await compileAsync(source, {
    fileName: "/repo/src/button.tsx",
    cssRoot: "/repo/src/app.css",
    sourceMap: true,
})
```

You can also pass inline CSS:

```ts
await compileAsync(source, {
    fileName: "/repo/src/button.tsx",
    cssSource: '@import "tailwindcss";',
})
```

When using either programmatic API directly, you are responsible for forwarding
`result.candidates` into your Tailwind CSS build. The Vite plugin handles this
automatically.

## Why CreateCompiledTailwindest Matters

The compiler makes nested variants easier to write because it can rebuild the
final Tailwind prefix from the object path. That is why compiler projects should
use [`CreateCompiledTailwindest`](/apis/createcompiledtailwindest) instead of
normal [`CreateTailwindest`](/apis/createtailwindest).

### Runtime Tailwindest

Runtime Tailwindest cannot assume a build-time compiler will normalize object
paths. The nested value must include the final Tailwind prefix:

```ts
type Tailwindest = CreateTailwindest<{
    tailwind: Tailwind
    tailwindNestGroups: TailwindNestGroups
}>

tw.style({
    backgroundColor: "bg-red-50",
    dark: {
        backgroundColor: "dark:bg-red-900",
        hover: {
            backgroundColor: "dark:hover:bg-red-950",
        },
    },
})
```

This is compatible with runtime-only Tailwindest, but the editor still has to
autocomplete `dark:` and `dark:hover:` inside leaf values.

### Compiler Tailwindest

Compiler Tailwindest already knows the object path. The value only needs the
raw utility:

```ts
type Tailwindest = CreateCompiledTailwindest<{
    tailwind: Tailwind
    tailwindNestGroups: TailwindNestGroups
}>

tw.style({
    backgroundColor: "bg-red-50",
    dark: {
        backgroundColor: "bg-red-900",
        hover: {
            backgroundColor: "bg-red-950",
        },
    },
})
```

The compiler produces the same effective classes:

```ts
"bg-red-50 dark:bg-red-900 dark:hover:bg-red-950"
```

### Practical Difference

| Topic               | `CreateTailwindest`                          | `CreateCompiledTailwindest`                                              |
| ------------------- | -------------------------------------------- | ------------------------------------------------------------------------ |
| Best for            | Runtime Tailwindest                          | `@tailwindest/compiler` projects                                         |
| Nested object keys  | Variant path                                 | Variant path                                                             |
| Nested leaf values  | Prefixed classes                             | Raw utilities                                                            |
| Editor autocomplete | Includes `dark:`, `hover:`, stacked prefixes | Focuses on utility names                                                 |
| Migration path      | Keep runtime API unchanged                   | Switch compiler projects to `CreateCompiledTailwindest`                  |
| Output result       | Runtime computes classes                     | Compiler lowers exact class-output calls and records Tailwind candidates |

The user-facing benefit is smaller authoring surface. You express state once in
the object path and keep each leaf focused on the utility you actually want. In
large design systems this reduces duplicated prefixes, improves autocomplete,
and makes deeply nested variants easier to scan.
