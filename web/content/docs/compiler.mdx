title: Compiler

> If you don't need `compiler`, you can just use normal tailwindest software.
>
> `compiler` provides great DX by resolve prefix writing problem and reduce runtime overhead using build-time calculation.

`@tailwindest/compiler` is the zero-runtime compiler for Tailwindest. It
evaluates supported `createTools()` calls during build and replaces them with
static Tailwind class strings, style records, or bounded lookup code.

Use it when you want Tailwindest authoring DX in source code, but optimized
Tailwind output in the browser bundle.

## What It Does

```tsx
const button = tw.style({
    display: "flex",
    backgroundColor: "bg-red-50",
    dark: {
        backgroundColor: "bg-red-900",
        hover: {
            backgroundColor: "bg-red-950",
        },
    },
})

button.class()
```

Compiled output is equivalent to:

```ts
"flex bg-red-50 dark:bg-red-900 dark:hover:bg-red-950"
```

Fully compiled call sites do not need Tailwindest styler runtime logic in the
browser.

## Install

```bash
pnpm add tailwindest tailwindcss @tailwindcss/vite vite
pnpm add -D @tailwindest/compiler
```

The compiler targets Vite and Tailwind CSS v4. Tailwind CSS and Vite remain
application-owned dependencies.

## Vite Setup

Place `tailwindest()` before `tailwindcss()`:

```ts filename="vite.config.ts"
import tailwindcss from "@tailwindcss/vite"
import { tailwindest } from "@tailwindest/compiler/vite"
import { defineConfig } from "vite"

export default defineConfig({
    plugins: [
        tailwindest({
            mode: "strict",
            debug: true,
            sourceMap: true,
        }),
        tailwindcss(),
    ],
})
```

The Vite integration returns two coordinated plugins:

- `tailwindest:transform` compiles JavaScript and TypeScript modules.
- `tailwindest:source` injects generated Tailwind candidates into CSS through
  Tailwind CSS v4 `@source inline()`.

This means development, debug, and production builds use the same compiler
manifest instead of relying on Tailwind to discover post-transform JavaScript.

## Authoring Types

Compiler-oriented projects should use `CreateCompiledTailwindest`. It keeps
variant prefixes in object keys and uses raw Tailwind utilities at leaf values.

```ts filename="tw.ts"
import {
    createTools,
    type CreateCompiledTailwindest,
    type CreateTailwindLiteral,
} from "tailwindest"
import type { Tailwind, TailwindNestGroups } from "./tailwind"

export type Tailwindest = CreateCompiledTailwindest<{
    tailwind: Tailwind
    tailwindNestGroups: TailwindNestGroups
    useArbitrary: true
}>

export type TailwindLiteral = CreateTailwindLiteral<Tailwind>

export const tw = createTools<{
    tailwindest: Tailwindest
    tailwindLiteral: TailwindLiteral
}>()
```

Nested variants should be written like this:

```ts
tw.style({
    backgroundColor: "bg-red-50",
    dark: {
        backgroundColor: "bg-red-900",
        hover: {
            backgroundColor: "bg-red-950",
        },
    },
})
```

Do not write prefixed leaf values in compiler-oriented source:

```ts
tw.style({
    dark: {
        backgroundColor: "dark:bg-red-900",
    },
})
```

The compiler accepts explicit-prefixed leaves for compatibility, but raw nested
leaves are the canonical form for `CreateCompiledTailwindest`.

## Strict and Loose Modes

`strict` mode is for CI and production builds. It fails when a Tailwindest call
cannot be compiled exactly.

```ts
tailwindest({ mode: "strict" })
```

`loose` mode is for incremental migration. Unsupported call sites stay in the
source as runtime fallbacks, while static candidates are still recorded for the
Tailwind manifest.

```ts
tailwindest({ mode: "loose", debug: true })
```

Use `strict` when you expect zero-runtime output. Use `loose` when a project is
still moving dynamic styling patterns into compiler-supported shapes.

## Options

```ts
tailwindest({
    include: [/src\/.*\.[jt]sx?$/],
    exclude: [/node_modules/],
    mode: "strict",
    debug: true,
    sourceMap: true,
    variantTableLimit: 256,
    cssEntries: [/src\/app\.css$/],
    scanRoots: ["src"],
})
```

| Option | Default | Description |
| --- | --- | --- |
| `include` | all JS/TS modules | Source id patterns eligible for compilation. |
| `exclude` | none | Source id patterns never compiled or used as CSS entries. |
| `mode` | `"strict"` | Unsupported-value policy: `"strict"` or `"loose"`. |
| `debug` | `false` | Writes `.tailwindest/debug-manifest.json`. |
| `debugFile` | `.tailwindest/debug-manifest.json` | Custom debug manifest path. |
| `sourceMap` | `false` | Emits source maps for exact replacements. |
| `variantTableLimit` | internal default | Maximum generated lookup table size for bounded variants. |
| `cssEntries` | Tailwind CSS imports | CSS ids that should receive `@source inline()` injection. |
| `scanRoots` | `["src"]` under Vite root | Directories scanned before the first CSS transform. |
| `merger` | `{ kind: "none" }` | Build-time class merge policy. |
| `collectStringLiteralCandidates` | `true` | Collects standalone candidate-like string literals from transformed modules. |

## Supported API Surface

The compiler targets the public `createTools()` API:

| API | Static output | Dynamic output |
| --- | --- | --- |
| `tw.style(obj).class()` | class string | fallback when unsupported |
| `tw.style(obj).style()` | style record | fallback when unsupported |
| `tw.style(obj).compose()` | composed styler metadata | fallback when unsupported |
| `tw.toggle(config).class(condition)` | selected class string | ternary |
| `tw.toggle(config).style(condition)` | selected style record | ternary object |
| `tw.rotary(config).class(key)` | selected class string | lookup |
| `tw.rotary(config).style(key)` | selected style record | lookup object |
| `tw.variants(config).class(props)` | class string | bounded lookup or fallback |
| `tw.variants(config).style(props)` | style record | bounded lookup or fallback |
| `tw.join(...)` | class string | partial fold or fallback |
| `tw.def(...)` | class string | partial fold or fallback |
| `tw.mergeProps(...)` | class string | fallback if any style is unknown |
| `tw.mergeRecord(...)` | style record | fallback if any style is unknown |

Compile eligibility is based on symbol provenance, not variable names. A
receiver named `tw` is not enough; the compiler must prove that it originates
from `createTools()`.

## Tailwind CSS Bridge

The compiler does not use Tailwind plugin `addUtilities`. Tailwindest generates
candidate class names for Tailwind CSS built-in utilities; it does not define
new CSS utilities.

Generated candidates are injected into CSS with Tailwind CSS v4 `@source
inline()`:

```css filename="app.css"
@import "tailwindcss";

/* tailwindest:start */
@source inline("bg-red-50 dark:bg-red-900 dark:hover:bg-red-950");
/* tailwindest:end */
```

When nested raw leaves would otherwise leak as top-level classes, the compiler
can also inject effective exclusions with `@source not inline()`.

## Debugging

Enable debug mode during integration:

```ts
tailwindest({
    mode: "strict",
    debug: true,
    sourceMap: true,
})
```

The debug manifest records:

- transformed files and hashes
- replacement spans
- generated text
- Tailwind candidates
- effective exclusions
- fallback status
- diagnostics

Inspect `.tailwindest/debug-manifest.json` when a class is missing from the CSS
output or a call unexpectedly falls back.

## Programmatic Compile

Build tools that cannot use the Vite plugin can call `compile()` directly:

```ts
import { compile } from "@tailwindest/compiler"

const result = compile(source, {
    fileName: "/repo/src/button.tsx",
    mode: "strict",
    sourceMap: true,
})

console.log(result.code)
console.log(result.candidates)
console.log(result.diagnostics)
```

When using `compile()` directly, you are responsible for forwarding
`result.candidates` into your Tailwind CSS build. The Vite plugin handles this
automatically.
