`@tailwindest/css-transformer` is a migration tool. It rewrites supported
Tailwind class-string patterns into Tailwindest object-style calls, while
preserving source code when exact conversion is not proven.

Use it when you are moving an existing Tailwind codebase from `className`,
`cn`, `clsx`, `classNames`, or `cva` strings into Tailwindest styles.

## Install

Run the transformer directly:

```bash
npx @tailwindest/css-transformer
```

Or install it for repeated local use:

```bash
pnpm add -D @tailwindest/css-transformer
```

The CLI opens an interactive prompt for target path, Tailwind CSS entry,
Tailwindest import name, import path, enabled walkers, dry-run mode, and output
mode.

## What It Converts

Input:

```tsx
import { cn } from "@/lib/utils"

export function Button() {
    return (
        <button
            className={cn(
                "flex items-center rounded-md bg-red-50 px-4 py-2",
                "dark:hover:bg-red-950"
            )}
        />
    )
}
```

Compiled-mode output:

```tsx
import { tw } from "~/tw"

const buttonButton = tw.style({
    display: "flex",
    alignItems: "items-center",
    borderRadius: "rounded-md",
    backgroundColor: "bg-red-50",
    paddingLeft: "px-4",
    paddingTop: "py-2",
    dark: {
        hover: {
            backgroundColor: "bg-red-950",
        },
    },
})

export function Button() {
    return <button className={buttonButton.class()} />
}
```

Runtime-mode output keeps prefixed nested leaves:

```ts
tw.style({
    dark: {
        hover: {
            backgroundColor: "dark:hover:bg-red-950",
        },
    },
})
```

## Output Modes

The transformer supports three output modes:

| Mode | Target | Leaf value |
| --- | --- | --- |
| `runtime` | `CreateTailwindest` | original token, such as `dark:hover:bg-red-950` |
| `compiled` | `CreateCompiledTailwindest` | raw utility, such as `bg-red-950` |
| `auto` | detected project mode | runtime unless strong compiler evidence exists |

Use `runtime` when you are migrating to normal Tailwindest runtime authoring.
Use `compiled` when the transformed source will be consumed by
`@tailwindest/compiler` and typed with `CreateCompiledTailwindest`.

## Recommended Mode Selection

```bash
# Safe default for mixed or unknown projects
npx @tailwindest/css-transformer --mode auto

# Runtime Tailwindest migration
npx @tailwindest/css-transformer --mode runtime

# Compiler-oriented migration
npx @tailwindest/css-transformer --mode compiled
```

`auto` switches to compiled output only from strong signals:

- a Tailwindest config declaring compiler output
- Vite config using `@tailwindest/compiler/vite`
- a precompile bridge using `createCompilerContext`
- source importing `CreateCompiledTailwindest`
- explicit CLI/API mode

Weak signals do not switch modes by themselves:

- `.tailwindest/debug-manifest.json`
- `@tailwindest/compiler` listed in `package.json`

Weak signals produce diagnostics and keep runtime output so existing
`CreateTailwindest` users are not broken by accident.

## Compiler-Oriented Setup

If you are migrating source for the zero-runtime compiler, define your
Tailwindest type with `CreateCompiledTailwindest`:

```ts filename="tw.ts"
import {
    createTools,
    type CreateCompiledTailwindest,
    type CreateTailwindLiteral,
} from "tailwindest"
import type { Tailwind, TailwindNestGroups } from "./tailwind"

export type Tailwindest = CreateCompiledTailwindest<{
    tailwind: Tailwind
    tailwindNestGroups: TailwindNestGroups
    useArbitrary: true
}>

export type TailwindLiteral = CreateTailwindLiteral<Tailwind>

export const tw = createTools<{
    tailwindest: Tailwindest
    tailwindLiteral: TailwindLiteral
}>()
```

Then transform with compiled output:

```bash
npx @tailwindest/css-transformer --mode compiled
```

Nested variant leaf values will be raw utilities:

```ts
tw.style({
    dark: {
        backgroundColor: "bg-red-900",
        hover: {
            backgroundColor: "bg-red-950",
        },
    },
})
```

## Runtime Tailwindest Setup

If you are not using the compiler, keep `CreateTailwindest`:

```ts filename="tw.ts"
import {
    createTools,
    type CreateTailwindest,
    type CreateTailwindLiteral,
} from "tailwindest"
import type { Tailwind, TailwindNestGroups } from "./tailwind"

export type Tailwindest = CreateTailwindest<{
    tailwind: Tailwind
    tailwindNestGroups: TailwindNestGroups
    useArbitrary: true
}>

export type TailwindLiteral = CreateTailwindLiteral<Tailwind>

export const tw = createTools<{
    tailwindest: Tailwindest
    tailwindLiteral: TailwindLiteral
}>()
```

Transform with runtime output:

```bash
npx @tailwindest/css-transformer --mode runtime
```

Nested variant leaves preserve the original prefixed class:

```ts
tw.style({
    dark: {
        hover: {
            backgroundColor: "dark:hover:bg-red-950",
        },
    },
})
```

## Supported Patterns

The transformer currently supports static class strings in:

- `className="..."`
- `className={"..."}`
- `cn("...")`
- `clsx("...")`
- `classNames("...")`
- `cva("...")`
- `cva(..., { variants: { ... } })` static string options

Dynamic arguments are preserved when the surrounding call can still be
represented safely.

```tsx
cn("flex px-4", isActive && "bg-red-500", props.className)
```

The static part can become a Tailwindest style constant, while dynamic arguments
remain in the generated `.class(...)` call.

## Conservative Fallback

The transformer does not rewrite code it cannot prove safe.

Common fallback cases:

- template literals with substitutions
- computed class strings
- unknown or ambiguous Tailwind utilities
- runtime-generated `cva` variant maps
- unsupported `compoundVariants` conversion
- helper imports still used elsewhere

Run dry-run first and inspect diagnostics before writing files.

## CLI Options

```bash
npx @tailwindest/css-transformer --mode auto --dry-run
```

| Option | Alias | Default | Description |
| --- | --- | --- | --- |
| `--css <path>` | `-c` | `tailwind.css` | Tailwind CSS entry used to initialize the resolver. |
| `--identifier <name>` | `-i` | `tw` | Tailwindest import identifier. |
| `--module <path>` | `-m` | `~/tw` | Tailwindest module import path. |
| `--dry-run` | `-d` | `false` | Preview without writing transformed files. |
| `--mode <mode>` | none | `auto` | Output mode: `auto`, `runtime`, or `compiled`. |

## Programmatic Usage

```ts
import { transform } from "@tailwindest/css-transformer"

const result = await transform(source, {
    resolver,
    outputMode: "compiled",
    projectRoot: process.cwd(),
    sourcePath: "/repo/src/Button.tsx",
    tailwindestIdentifier: "tw",
    tailwindestModulePath: "~/tw",
    walkers: ["cva", "cn", "classname"],
    config: {
        objectThreshold: 2,
    },
})

console.log(result.code)
console.log(result.diagnostics)
```

You normally do not need the programmatic API unless you are building custom
migration tooling.

## Safety Model

The transformer uses Collect -> Reverse Execute:

1. Parse source with `ts-morph`.
2. Collect supported transform targets.
3. Analyze static class strings.
4. Apply replacements from the end of the file to the beginning.
5. Apply import edits once.
6. Return transformed code and diagnostics.

This avoids stale AST ranges and keeps unrelated source code untouched.
